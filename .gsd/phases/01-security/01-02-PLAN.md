<objective>
Eliminate client-controlled pricing by computing unit price server-side for all order creation and merge operations. Client-supplied `unitPrice` values must be ignored — prices come from `card_type_pricing` table only.

Purpose: Prevent underpayment — a manipulated client request setting price to $0.00 must NOT result in a $0 order.
Output: Both the create-order path and merge-order path in `api/orders/+server.ts` use server-computed prices.
</objective>

  <action>

    **Step 1: Add the imports.**
    Near the top of the file, add these two imports (from different modules):
    ```typescript
    import { fetchPrices } from '$lib/server/pricing';
    import { getCardPrice } from '$lib/utils';
    ```
    Note: `fetchPrices` is in `$lib/server/pricing`, but `getCardPrice` is in `$lib/utils` — they are in different files.

    **Step 2: Resolve prices once per request.**
    In the POST handler, BEFORE the loop that builds order items to insert, add:
    ```typescript
    const prices = await fetchPrices(locals.supabase);
    ```

    **Step 3: For each item in the create path (around line 307)**, replace:
    ```typescript
    unit_price: item.unitPrice,
    ```

    With server-side price lookup:
    ```typescript
    // Fetch card type for authoritative price
    const { data: cardRow, error: cardErr } = await locals.supabase
      .from('cards')
      .select('card_type')
      .eq('id', item.cardId)
      .single();
    if (cardErr || !cardRow) throw error(400, `Card not found: ${item.cardId}`);
    const serverPrice = getCardPrice(cardRow.card_type, prices);
    ```

    Then use `serverPrice` instead of `item.unitPrice` in the insert object:
    ```typescript
    unit_price: serverPrice,
    ```

    If multiple items are inserted in a loop, fetch the card type inside the loop for each item (or batch: fetch all cardIds at once using `.in('id', items.map(i => i.cardId))` and build a map). The batch approach is preferred if items is an array.

    **Important:** Do NOT remove `unitPrice` from the request type or destructuring — its presence in the payload is fine, it just must not be used for DB writes.
